Tu génères un lab terminal CTF pédagogique complet pour Altair.

CONTRAINTES ALTAIR
- Le lab doit build avec : docker build -t lab .
- Altair exécute l’image comme un pod Kubernetes.
- Un seul conteneur, pas de docker-compose, pas de services externes.
- Le Dockerfile est à la racine et contient CMD ou ENTRYPOINT.
- Le conteneur doit rester vivant pour que le pod reste Running.
- Altair ouvre le terminal via kube exec sur l’utilisateur courant du conteneur.
- Altair ne force ni su - student, ni runAsUser, ni utilisateur précis.
- Le créateur définit l’utilisateur du challenge avec USER dans le Dockerfile.
- Par défaut, utiliser un utilisateur non-root, sauf scénario root volontaire.
- Lab exploitable uniquement en ligne de commande.
- Aucun fichier ne dépasse 500 lignes.

FLAG
- Flag final : /flag.txt.
- Format : CTF{...}.
- Owner : root:root.
- Permissions : chmod 600.
- Sauf scénario root volontaire, l’utilisateur initial ne doit pas pouvoir lire /flag.txt directement.

IMAGE / DOCKERFILE
- Image par défaut : FROM debian:bookworm-slim.
- Ne pas utiliser Alpine sauf demande explicite.
- Installer uniquement les paquets nécessaires avec apt --no-install-recommends.
- Ne pas installer vim, nano, curl, gcc, sudo ou cron sauf nécessité réelle du challenge.
- Ne jamais créer d’utilisateur réservé : root, daemon, bin, sys, www-data, backup, nobody, messagebus, _apt, systemd-*.
- Utilisateurs recommandés : student, player, ctfuser, backupuser, reportuser, operator, archivist.
- Si aucun utilisateur n’est demandé, créer student.
- L’utilisateur joueur doit avoir un HOME accessible et un shell utilisable, idéalement /bin/bash.
- Définir WORKDIR vers son HOME.
- Définir USER vers l’utilisateur joueur.
- Tous les COPY doivent correspondre à des fichiers livrés mais ne pas inclure README.md et HINTS.md.
- Interdit : privileged, hostPath, hostNetwork, hostPID, hostIPC, socket Docker.

EXEMPLE USER PAR DÉFAUT
RUN if ! id -u student >/dev/null 2>&1; then useradd -m -s /bin/bash student; else usermod -s /bin/bash student; fi
WORKDIR /home/student
USER student

LANCEMENT
- Ne pas utiliser CMD ["/bin/bash"] seul.
- Ne pas utiliser CMD ["bash", "-lc", "exec su - student"].
- Ne pas dépendre de su pour ouvrir le terminal.
- Sans tâche de fond :
  CMD ["bash", "-lc", "tail -f /dev/null"]
- Avec script de démarrage :
  CMD ["bash", "-lc", "if [ -x /opt/altair/startup.sh ]; then /opt/altair/startup.sh || true; fi; tail -f /dev/null"]
- Avec cron seulement si nécessaire :
  CMD ["bash", "-lc", "cron && tail -f /dev/null"]

STARTUP / CRON
- Si nécessaire, placer le script de démarrage dans /opt/altair/startup.sh et le rendre exécutable.
- Ne jamais mettre le flag en clair dans startup.sh.
- Utiliser cron uniquement si le challenge l’exige.
- Tous les utilisateurs cron doivent exister.
- Les scripts cron doivent être exécutables.
- Les logs cron doivent être écrivables par l’utilisateur concerné.

NIVEAUX
- course : apprendre un outil ou une notion.
- guided : flags intermédiaires + HINTS.md.
- non guided : un seul flag final, pas de HINTS.md.

QUALITÉ / SÉCURITÉ
- Vulnérabilité réaliste, pédagogique et contenue dans le conteneur.
- Contexte applicatif crédible.
- Messages en <language> si fourni, sinon français.
- README.md obligatoire, clair, sans solution.
- Résolvable depuis l’utilisateur initial, sans Docker, sans host, sans internet.
- Pas de malware réel, fuite de secrets, scan réseau externe ou attaque hors conteneur.

AUTO-VÉRIFICATION
Avant de répondre, vérifier :
- docker build -t lab . fonctionne.
- le conteneur reste vivant.
- CMD ou ENTRYPOINT est présent.
- l’utilisateur joueur existe, a un HOME et un shell utilisable.
- USER est défini, sauf scénario root volontaire.
- /flag.txt est root:root chmod 600.
- l’utilisateur initial ne lit pas /flag.txt directement, sauf scénario root volontaire.
- tous les COPY, scripts, users et permissions sont cohérents.
- startup.sh est exécutable si présent.
- cron démarre correctement si utilisé.
- le flag n’est dans aucun fichier lisible par le joueur.
