ALTAIR TERMINAL CONSTRAINTS

- The lab must build with:
  docker build -t lab .
- Altair runs the image as a Kubernetes Pod.
- Single container.
- No external services.
- No internet at runtime.
- The lab is exploitable only via command line.
- Altair opens the terminal via kube exec on the container's current user.
- Altair does not force su - student, runAsUser, or a specific user.
- The Dockerfile defines the challenge user with USER.
- By default, use a non-root user, unless the scenario explicitly requires root.
- The container must stay alive so the pod remains Running.
- Forbidden: privileged, hostPath, hostNetwork, hostPID, hostIPC, Docker socket.
- The lab must remain self-contained in the container: no off-platform attacks, no external scanning, no real malware.

EXPECTED TERMINAL STRUCTURE

- The terminal lab must contain at minimum:
  - Dockerfile
  - entrypoint.sh
  - README.md
- entrypoint.sh must be at the project root.
- Do not use /opt/altair/startup.sh or scripts/startup.sh as the main entry point.
- All files referenced by COPY, CMD, ENTRYPOINT or scripts must exist.
- Do not produce unused or dead files.

IMAGE / DOCKERFILE

- Default image: debian:bookworm-slim.
- Do not use Alpine unless explicitly requested.
- Install only necessary packages with apt --no-install-recommends.
- Do not install vim, nano, curl, gcc, sudo or cron unless genuinely needed by the challenge.
- Never create reserved users:
  root, daemon, bin, sys, www-data, backup, nobody, messagebus, _apt, systemd-*.
- Recommended users:
  student, player, ctfuser, backupuser, reportuser, operator, archivist.
- If no user is requested, create student.
- The player user must have an accessible HOME and a usable shell, ideally /bin/bash.
- Set WORKDIR to the player user's HOME.
- Set USER to the player user, unless the scenario explicitly requires root.
- Do not copy README.md or HINTS.md into the image unless genuinely needed by the challenge.
- The Dockerfile must copy and use entrypoint.sh:

COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
CMD ["/entrypoint.sh"]

ENTRYPOINT

- All startup logic must be in entrypoint.sh.
- Do not use CMD ["/bin/bash"] alone.
- Do not depend on su to open the terminal.
- Do not put a long bash command directly in CMD.
- entrypoint.sh must keep the container alive.
- If no service is running, end with:

exec tail -f /dev/null

- If services start, entrypoint.sh must:
  - create necessary directories;
  - launch only services useful to the challenge;
  - write logs to consistent paths;
  - not contain the flag in plaintext;
  - end with a command that keeps the container alive.
- All scripts called by entrypoint.sh must exist and be made executable.

PORTS / SERVICES

- If USER is non-root, do not start services on ports below 1024.
- Ports 21, 22, 80 and 443 are forbidden with a non-root user, unless the scenario coherently requires root.
- Recommended ports:
  - HTTP: 3000, 8000 or 8080
  - Mock FTP: 2121
  - Mock SSH: 2222
  - Custom service: 9000+
- Ports in README, scripts, flag and commands must match the ports actually started.
- Never ask in README.md to scan a port that is not started.
- Never put in the flag ports different from the ones actually used.
- If the user explicitly requests privileged ports with a non-root user, adapt to coherent non-privileged ports or return ERROR: <reason>.

FLAG

- The final flag is stored in /flag.txt.
- The flag can be created in the Dockerfile or in a generation script if those files are never accessible to the player.
- The final flag must never be placed in a file accessible to the player at runtime, except /flag.txt with the intended permissions.
- Never put the final flag in README.md, HINTS.md, player HOME files, static files, player-readable logs or public endpoints.
- Do not make the flag impossible to obtain.
- For a course lab without privilege escalation, the flag can be readable by the player user after the walkthrough.
  Example:
  chown root:<player user> /flag.txt
  chmod 640 /flag.txt
- For a guided or unguided challenge with real exploitation, use instead:
  chown root:root /flag.txt
  chmod 600 /flag.txt
- If /flag.txt is root:root chmod 600, a real, pedagogical and testable mechanism must allow the player to access it.
- Never write sudo cat /flag.txt unless sudo is installed, configured and is an explicit part of the challenge.

README.md

- README.md must explain:
  - the lab context;
  - the pedagogical objective;
  - initial hints;
  - useful constraints;
  - the expected outcome without revealing the flag.
- README.md must not give the complete solution.
- README.md must not require Docker, Kubernetes, the host, internet, or an external service.
- README.md must not mention a tool that is not installed.
- README.md must not mention a port, service, user or file that does not exist.
- README.md must not require sudo if sudo is not installed and configured.
- For a course level, do not add unnecessary privilege escalation if the objective is only to learn a tool.

CRON

- Cron is forbidden unless genuinely needed by the challenge.
- If cron is needed:
  - install cron explicitly;
  - create all users used by cron;
  - make cron scripts executable;
  - write logs to accessible paths;
  - start cron from entrypoint.sh;
  - keep the container alive after startup.
