ALTAIR WEB CONSTRAINTS

- The lab must work locally with:
  docker build -t lab .
  docker run -p <app_port>:<app_port> lab
- If <app_port> is not provided, use 3000.
- In production, Altair runs the image as a Kubernetes Pod with lab_delivery="web" and app_port=<app_port>.
- Single container.
- No external services.
- No internet at runtime.
- The Altair web runtime exposes a single public port.
- The Dockerfile is at the root and contains EXPOSE <app_port> and CMD or ENTRYPOINT.
- The web server must stay in the foreground.
- The container must not terminate after startup.
- Forbidden: privileged, hostPath, hostNetwork, hostPID, hostIPC, Docker socket.
- The lab must be self-contained in the container, with no off-platform attacks.

WEB PORT AND HOST

- The effective port is <app_port> if provided, otherwise 3000.
- The Dockerfile must expose only the effective port.
- The main web server must listen on all container network interfaces.
- Use 0.0.0.0 by default.
- Never use localhost, 127.0.0.1 or ::1 as the main server listening host.
- Note: the listening host is not the public URL of the lab. The public URL is managed by Altair via the proxy.
- The host must not be made configurable by the user, unless there is an explicit technical need validated by Altair constraints.
- All public web traffic must go through the effective port.
- Never expose multiple public ports.
- Never require a second docker run -p.

ALTAIR ROUTING / SUB-PATH

- The lab is accessed via a URL of the form /web/<container_id>/.
- The application must work behind this sub-path.
- Use relative paths for assets, links, redirections, and API calls.
- Avoid absolute paths like /style.css, /script.js, /api/login, /login.
- Prefer:
  ./style.css
  ./script.js
  ./api/login
  new URL("api/login", window.location.href)
- Do not hardcode localhost, 127.0.0.1, 0.0.0.0, http://localhost:<app_port>, http://localhost:3000 or a fixed domain in the frontend.
- The frontend must never directly call a fixed host or port.
- Redirections must remain relative, e.g. ./dashboard instead of /dashboard.
- If the app uses React/Vite/SPA, configure the build to work with a relative path.
- If an HTML page references static files, add if useful:
  <base href="./">

INTERNAL PORTS AND SERVICES

- If the scenario requires multiple internal services, run them in the same container behind an internal reverse proxy.
- In that case, Nginx/Caddy/equivalent must listen on 0.0.0.0:<app_port> and stay in the foreground.
- Internal services must not be exposed publicly on other ports.
- Internal ports, README, scripts and configuration must remain consistent.

- /flag.txt must never be served as a static file.
- /flag.txt must never be placed in public/, static/, assets/, dist/ or any folder exposed by the web server.

- Never put the final flag in the HTML, frontend JavaScript, CSS, README.md, HINTS.md, player-readable logs or unprotected public endpoints.

STARTUP

- CMD must launch the main web server in the foreground.
- Valid examples:
  CMD ["node", "server.js"]
  CMD ["python", "app.py"]
  CMD ["npm", "start"]
- For Express:
  app.listen(APP_PORT, "0.0.0.0")
  with APP_PORT equal to the effective port.
- For Flask:
  app.run(host="0.0.0.0", port=APP_PORT)
- For FastAPI/Uvicorn:
  uvicorn app:app --host 0.0.0.0 --port <app_port>
- For Vite preview:
  vite preview --host 0.0.0.0 --port <app_port>
- Do not use CMD ["/bin/bash"] alone.
- Do not launch the main web server in the background and then only do tail -f /dev/null.
- An entrypoint.sh is allowed only if multiple internal processes are needed.
- If entrypoint.sh is used, it must launch the main web server in the foreground or use an explicitly justified minimal supervisor.
